Areas de Git

Working Directory

El area de trabajo corresponde a la carpeta donde iniciamos el repositorio y tambien donde vamos a estar realizando los cambios de nuestro proyecto. Practicamente no necesitamos utilizar ningun comando, salvo en muy pocas situaciones.

  • dir || ls muestran el contenido del directorio actual
  • mkdir <name crea un directorio con el nombre indicado
  • cd <path permite cambiar de directorio, accediendo a la ruta especificada
  • git init Inicializa un repositorio de gir, creando la carpeta .git
  • rm -rf .git Elimina el repositorio, manteniendo los archivos, los parametros -rf afectan carpetas y lo hacen de manera forzada

Staging Area (INDEX)

El area de control de cambios se utiliza para agregar los archivos a los que queremos realizarles seguimiento, de esta manera GIT nos puede notificar cuando estos fueron modificados y el estado de los mismos; para lograr esto, se realiza un Snapshot (captura de pantalla) del codigo del archivo. Los archivos que no sean agregados al INDEX, no podran ser reestablecidos y tampoco se llevara contro de los mismos.

  • git status compara el contenido del INDEX con el contenido del Working Directory y devuelve el estado de los archivos
    • untracked archivo al que no se le esta haciendo seguimiento
    • staged archivo agregado al INDEX pero al que nunca se le realizo un commit
    • modified archivo cuyo contenido difiere de lo registrado en el INDEX
    • commited archivo cuyos cambios ya fueron confirmados
  • git add <file se utiliza para agregar un archivo o conjunto de archivos al Staging Area (INDEX)
  • git reset <file --cached quita del INDEX un archivo agregado por primera vez
  • git restore --staged <file quita del INDEX las modificaciones realizadas sobre un archivo

Repository

Corresponde al lugar donde se almacenan las capturas en formato BLOB (Binary Large Object File) generando un numero de HASH utilizando la informacion de nombre usuario, correo, titulo del mensaje, registro de cambios, ademas de la fecha y la hora en la que se realiza la confirmacion. Esto es necesario para registrar la relacion que existe entre dicho commit y las confirmaciones de cambios anteriores.

  • git commit genera la confirmacion de los cambios almacenados en el INDEX
  • git diff <ref compara el contenido del Working Directory con el commit que se indica en la referencia
  • git blame <file muestra en cada linea de codigo los datos del autor, fecha y hora de confirmacion del archivo seleccionado
  • git checkout <ref permite el desplazamiento dentro del historial, moviendonos entre los commits referenciados

Apuntadores GIT

HEAD

Apuntador movil que permite al usuario desplazarse entre los commits mediante referencias como por ejemplo: numero de HASH, referencias relativas como HEAD~2 o HEAD^, branches, tags, etc... gracias al comando checkout.

BRANCH

Apuntadores dinamicos que siempre corresponden al ultimo commit de una linea de tiempo particular y se utiliza para poder volver cuando revisamos confirmaciones anteriores, debido a que los commits solamente apuntan hacia sus ancestros porque son dirigidos y aciclicos.

TAG

Apuntador estatico que se utiliza para el versionado de nuestro proyecto, normalmente se asignan nombres como vn.n para identificar el numero de version principal y compilacion. Los mismos deben subirse por separado al repositorio remoto con el que trabajemos.
